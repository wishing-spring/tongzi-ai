# 源卦 · 三视角审视

## 一、工匠视角

### 要造什么

一个 28-bit 种子，按需产出任意层数的卦元。

```
源卦 = 固定 28-bit F₂ 模式

produce(源卦, k) → k 爻的卦
  k ≤ 28: 取源卦前 k bit
  k > 28: 源卦 28bit + φ 填充剩余 bit
```

工程上能做到。纯位截取 + φ 追加。零依赖、确定性。

### 实现难度

| 功能 | 难度 | 说明 |
|:--|:--|:--|
| produce(k) | 低 | 位截取 |
| 同一源卦的所有产出共享前缀 | 低 | 结构保证 |
| 多源卦实例 | 低 | 独立 bit 模式 |
| 产出卦可溯源至源卦 | 低 | 前缀匹配即可 |
| 环境感知生成（后续） | 中 | 需要定义"环境"是什么 |

### 不确定点

扩张时(k > 28)用什么填充？φ 是固定的——所有源卦扩张出的尾部都一样。这意味着：
- 两个不同源卦产出的 30 爻卦，前 28 位不同，后 2 位相同
- 尾部差异为零——两个卦组的"深层"缺乏区分度

可能需要：填充规则与源卦自身 bit 模式挂钩，而非全局 φ。

## 二、批判视角

### 弱联系

源卦与产出卦的唯一联系是**共享前缀**。它是一个结构连接，不是功能连接。

```
源卦: 0xA3F5...
子卦(k=5): 0b10100 = 源卦前5bit

"联系" = 子卦是源卦的一个投影。仅此而已。
```

这意味着：如果两个源卦的前 5 bit 碰巧相同，它们产出的 5 爻子卦完全相同。两个源卦的"个性"在浅层无法区分。

### 截断丢信息

从 28 bit 截到 3 bit，丢掉 25 bit 信息。剩下的 3 bit 还能代表这个源卦吗？

```
源卦 A: 0x1234567 → 前3bit: 0b001
源卦 B: 0x0012345 → 前3bit: 0b000
```

能。只要这两个源卦的前 3 bit 真不同。但如果不问——如果只差在尾部——浅层子卦就毫无区分。**浅层卦的区分度完全取决于源卦高位 bit 的多样性。**

### 源卦自己从哪来

源卦是"孤零零"的本体——但谁生成源卦？如果手动指定，系统不自主。如果 φ 自动生成，源卦之间缺少有意义的差异。

### 源卦的"能力边界"

用户说源卦应该"相对智能"——按指令增减爻。但目前的设计只是位截取。这不是智能，是机械规则。真正的"智能"需要：
- 根据上下文决定截取哪些位（不只是前 k bit）
- 或者根据需求动态选择填充规则

当前设计是**确定性的、无状态的、不感知上下文的**。这是刻意为之还是需要改进？

## 三、整成视角

### 边界

```
源卦 负责: 产出卦元（任何层数）
源卦 不负责:
  - 卦元怎么用（编码层的事）
  - 卦元之间怎么碰撞（碰撞层的事）
  - 卦元代表什么意义（上层的事）
  - 产出后的卦的行为（环境的事）
```

源卦是**供应者**，不是**控制者**。

### 源卦与其他部件的关系

```
源卦₁ ──→ 卦元A,B,C... ──→ 编码层
源卦₂ ──→ 卦元D,E,F... ──→ 记忆层
源卦₃ ──→ 卦元G,H,I... ──→ 输出层

不同源卦的产出之间通过碰撞产生联系。
源卦之间不直接通信。
```

### 多源卦的必要性

如果只有一个源卦，整个系统的所有卦元共享同一个前缀结构。这如同一个文明只用一个字母表——能工作，但缺乏多样性。

多源卦意味着：
- 编码层可用一套源卦（符号卦组）
- 记忆层可用另一套（轨迹卦组）
- 每套内部卦元有谱系关系，套与套之间通过碰撞交互

### 核心问题

源卦的设计解决了一个真问题（"卦从哪来"），但没有解决**编码问题**（"卦代表什么"）。

后者需要另一个部件。

---

## 总结

| 可以开工 | 需要先定 |
|:--|:--|
| produce(k) 位截取 | 扩张填充规则（φ or 源卦自身模式） |
| 多源卦实例 | 源卦自身的生成方式（手动 or φ） |
| 卦元溯源 | 源卦之间的联系是否需要 |
| 前缀保证结构关系 | 是否接受"浅层区分度依赖高位多样性" |

### 一句话

源卦是一个**确定性的、自包含的卦元供应器**。能造。但"联系"是位重叠而非功能关系，"智能"当前是机械规则而非自适应。这两个点如果要升级，需要你提供方向。
