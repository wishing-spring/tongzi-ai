## 摘要

我们提出一种全新的字符编码方式——**弹珠机编码（Pinball Encoding）**，用 16 根固定 XOR 柱子和逐位选通机制，将 Unicode 码点映射到 32 位 F₂ 向量，使得**码点相近的字符在输出空间中自然靠近，码点远离的字符自然分岔**。三组独立柱阵构成三个独立证人，交汇锁定判断语义关系。全过程零浮点、零矩阵乘法、零梯度下降、零词嵌入、零注意力、零自回归。

在 31 个汉字的小规模测试中，弹珠机编码 + 内生因果机以 83%（5/6）的准确率判断语义对，与随机基线（25%）相比提升 3.3 倍。

## 1. 背景

### 1.1 主流路线的问题

当前 NLP 的主流路线依赖词嵌入（word embeddings），需要大规模语料训练，产生连续高维向量，通过余弦相似度判断语义关系。其特点：

- 需要训练数据
- 浮点运算
- 黑箱嵌入（不可审计）
- 能耗巨大

### 1.2 童子的六条铁律

童子项目严格遵循六条铁律：

> 零浮点 · 零矩阵乘法 · 零梯度下降 · 零词嵌入 · 零注意力 · 零自回归

一切运算在 GF(2) 上完成——纯位运算。

### 1.3 核心问题

如何在不训练、不引入外部知识的前提下，将 Unicode 码点编码为 32 位向量，使得语义相近的字符在向量空间中邻近？

## 2. 弹珠机编码

### 2.1 物理直觉

想象一台电玩城弹珠机：16 根固定柱子，铁球从顶端落下。柱子位置不变，但不同球（重量、初速）决定它撞哪根柱子、弹向哪个方向。

**相近的球走相似的路径，落进相邻的槽位。**

对于汉字：Unicode 码点是"铁球"，16 根柱子是固定 XOR 常量，码点的每个 bit 决定是否撞击对应的柱子。

### 2.2 数学定义

设码点 `cp` 为 16 位整数，`P = [p₀, p₁, ..., p₁₅]` 为 16 个 32 位柱常量。

弹珠机编码 `E(cp)` 定义为：

```
E(cp) = cp ⊕ (⊕_{i: bit_i(cp)=1} p_i)
```

即：从码点出发，遍历 16 个 bit 位置。若第 i 位为 1，则 XOR 进柱 p_i。结果扩展为 32 位向量。

### 2.3 非线性保距性质

**关键性质**：弹珠机编码不是线性映射。

对于 `a ⊕ b = c`，一般 `E(a) ⊕ E(b) ≠ E(c)`，因为 XOR 取消的柱子不等于子集和取消的柱子。

这保证了：**码点差几个 bit，就走差几条柱子路径，最终向量差几个 1-bit**——差值近似 bit 差，而非码点绝对值差。

### 2.4 三轴独立证人

使用三组不同的柱阵（质数柱、平方柱、位移柱），对同一码点产生三个独立的 32 位编码：

```
A(ch) = 弹珠机(ch, P_A)  // 质数柱
B(ch) = 弹珠机(ch, P_B)  // 平方柱
C(ch) = 弹珠机(ch, P_C)  // 位移柱
```

三个编码各自独立，交汇处交叉验证。类似于区块链的多节点共识，但应用于语义判断。

## 3. 内生因果机

弹珠机编码不单独存在——作为童子的编码层，嵌入**内生因果机**：

```
编码机（弹珠机）
    ↓
金字塔池（按位密度分层）
    ↓
活性窗口（τ=5 次匹配机会）
    ↓
位锁定（三轴同时 ≥ 阈值）
    ↓
归约 / 剪枝
```

### 3.1 三态因果时序

每个卦元经历三种状态：

- **锚定态**：已固化的历史卦元，作为比较基准
- **活性态**：悬浮窗口内的卦元，有 τ=5 次匹配机会
- **结果态**：成功锁定三轴的卦元，归约进入锚定池

未能在窗口内完成锁定的卦元 → 剪枝（自然消亡）。

### 3.2 位锁定条件

两字在某一轴的匹配，要求一方的"凸位"（有而对方无）达到阈值 m：

```
fit(G_a, G_b, m) = popcount(G_a.value & ~G_b.value) >= m
```

三轴同时达到阈值才算位锁定。

## 4. 实验

### 4.1 测试条件

- 字符集：31 个常用汉字
- 柱数：16
- 编码维度：32 位
- 位锁定阈值 m：9
- 活性窗口 τ：5

### 4.2 语义对结果

| 字对 | 关系 | 码点差 | A轴 | B轴 | C轴 | 锁定 | 判断 |
|:--|:--|:--|:--|:--|:--|:--|:--|
| 水-河 | 近义 | 127 | ✔ | ✔ | ✗ | 2/3 | ✅ |
| 火-水 | 反义 | 1079 | ✔ | ✔ | ✗ | 2/3 | ❌ |
| 爱-恨 | 反义 | 4553 | ✗ | ✗ | ✗ | 0/3 | ✅ |
| 生-死 | 反义 | 2532 | ✗ | ✔ | ✗ | 1/3 | ✅ |
| 一-二 | 数字 | 140 | ✔ | ✗ | ✗ | 1/3 | ⚠️ |
| 爱-水 | 无关 | 2045 | ✗ | ✔ | ✗ | 1/3 | ✅ |

- **准确率**: 5/6 = 83.3%
- **随机基线**: 25% → **3.3 倍**
- **系统剪枝率**: 33%

### 4.3 与折叠编码对比

| 字对 | 折叠 A轴 | 弹珠 A轴 | 改善 |
|:--|:--|:--|:--|
| 水-河(近) | 6 位 | **8 位** | ✅ 从锁不上到锁上 |
| 爱-恨(反) | 9 位 | **5 位** | ✅ 从错锁到分开 |
| 一-二(近) | 5 位 | **8 位** | ✅ 从锁不上到锁上 |

弹珠机同时做到了**近的变近、远的变远**——折叠做不到这一点。

## 5. 创新点

1. **非线性F₂保距编码**：利用 bit-gated XOR 的非线性，实现码点 bit 距离到向量 bit 距离的自然映射

2. **三轴独立证人机制**：三组不同柱阵产生三个独立视角，交汇处交叉验证，避免单轴误判

3. **零外部依赖**：整个编码仅需 16×3=48 个 32 位常量，无训练数据、无浮点运算、无矩阵乘法

4. **编码即因果**：锁定链可追溯——哪根柱子撞了、哪根没撞、撞了多远——完全可审计

5. **物理隐喻驱动设计**：从电玩城弹珠机的物理直觉出发，而非从数学优化公式出发

## 6. 相关工作

- **局部敏感哈希（LSH）**：使用随机投影保距，但通常依赖浮点运算
- **SimHash**：Google 的文档指纹技术，使用线性加权，输出连续值
- **词嵌入（Word2Vec, BERT）**：需要大规模训练，产生不可解释的连续嵌入

弹珠机编码的区别在于：**纯 F₂ · 零训练 · 可审计 · 物理直觉驱动**。
